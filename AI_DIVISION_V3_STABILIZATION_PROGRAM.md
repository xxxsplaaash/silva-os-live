# AI Division v3 Stabilization Program

## Purpose

This program defines how to make the current live AI Division product smaller, truer, more reliable, and more useful without casually rewriting the legacy shell. The goal is not to make v3 “future perfect.” The goal is to make v3 trustworthy, disciplined, and operationally coherent enough to support real daily studio use while v4 is designed separately.

## Governing principle

v3 is the live operating product. It must be stabilized around product truth, not expanded around ambition. Every action in this program should make the system:

- clearer
- more authoritative
- less redundant
- more durable
- easier to trust
- easier to evolve

## What v3 is responsible for

v3 should be responsible for one thing above all else:

**Helping an internal operator direct, generate, review, and preserve character-consistent creative work inside one disciplined studio environment.**

That means v3 should prioritize:

- Studio Pulse direction
- prompt creation
- identity lock and reference management
- output review
- planning linkage
- continuity support

It should not try to fully embody every future platform ambition.

## What v3 is not responsible for

v3 is not the place to fully solve:

- multi-agent orchestration
- hosted/cloud-first system memory
- mature analytics intelligence
- broad provider abstraction layers
- external platformization
- every speculative system module

Those belong to v4 unless a narrow version is required to support the core v3 workflow.

## Stabilization outcomes

v3 is considered stabilized when:

1. The product has a clear governing workflow.
2. The navigation reflects real module priority instead of patch order.
3. The live runtime has one clear source of truth strategy.
4. Studio Pulse acts like a real operating surface, not a decorative smart page.
5. Identity lock, assets, Home System, and review are structurally tied to generation quality.
6. Prompts, outputs, review findings, and planning artifacts have traceable lineage.
7. Shell modules no longer compete visually with core workflows.
8. Operators can tell which surfaces are authoritative, which are support surfaces, and which are admin-only.

## v3 constraints

These constraints should be enforced throughout the stabilization program:

- Keep the current live shell as the baseline unless a change is clearly safer than leaving it alone.
- No broad rewrite.
- No large route migration unless a specific live blocker demands it.
- No new first-class pages unless they directly strengthen the core workflow.
- No intelligence theatrics that are not backed by deterministic rules, real data, or durable review memory.
- No design churn that is not tied to hierarchy, trust, speed, or consistency.

## Program structure

## Workstream 1: Product boundary reset

### Objective

Reduce ambiguity about what AI Division is for in v3.

### Required decisions

- `keep` Studio Pulse as the command surface
- `keep` Generator, Library, Gallery, Assets, Planner, Character System
- `formalize` Home System as a continuity support layer
- `merge` Workflow SOP into guidance or planning
- `merge` Content Ideas into planning or campaign preparation
- `merge` Campaigns into a lighter planning-and-lineage role unless real object relationships are added
- `merge` Team, Providers, Analytics, and Dev/Admin into a system/admin tier

### Execution actions

- Reclassify every live surface as `core`, `support`, `system`, or `experimental`
- Remove first-class nav parity from low-maturity system surfaces
- Update language so module names do not overstate maturity
- Define a single “what this product does” statement used across core surfaces

### Exit criteria

- A new operator can understand the product’s job in under 2 minutes
- The main nav clearly distinguishes core work from support and system work
- No module in the main operator path feels like a dressed-up placeholder

## Workstream 2: Source of truth and state discipline

### Objective

Stop the live product from feeling like multiple partial systems stacked together.

### Required decisions

- `formalize` one operational source of truth policy for v3
- `keep` local-first behavior where needed for stability
- `formalize` what SQLite owns versus what remains client-owned in v3
- `delay` disconnected Supabase ambitions until v4 unless a narrow path is explicitly activated

### Execution actions

- Define authoritative ownership for:
  - prompts
  - gallery outputs
  - planner posts
  - character state
  - relationships
  - review events
  - assets
  - continuity data
- Remove silent dual-ownership assumptions
- Make export/migrate paths explicit support tools, not hidden backup logic
- Treat every new persisted object as a contract, not an ad hoc patch addition

### Exit criteria

- Every core artifact has a clear owner
- Operators and developers can answer “where does this live?” unambiguously
- Runtime state no longer feels patch-grown and accidental

## Workstream 3: Workflow closure

### Objective

Turn adjacent modules into one compounding operating loop.

### Core v3 loop

1. Choose character and working context
2. Lock identity and relevant references
3. Generate or refine prompt
4. Produce output
5. Review output quality and drift
6. Save findings
7. Promote useful results into planning or reusable assets

### Execution actions

- `formalize` prompt record -> output record linkage
- `formalize` output record -> review event linkage
- `formalize` review event -> planner/campaign implication linkage
- `formalize` asset usage and continuity context on outputs where relevant
- Add “what happened next” thinking to planner and review surfaces

### Exit criteria

- Prompts do not die as isolated text
- Outputs do not die as isolated gallery cards
- Review findings visibly influence future decisions
- Planning becomes downstream from real work, not a separate decorative area

## Workstream 4: Studio Pulse operating-surface upgrade

### Objective

Make Studio Pulse the actual control layer of v3.

### Studio Pulse should do in v3

- answer direction questions
- identify weak continuity or asset coverage
- point out review debt
- recommend next actions
- route the operator toward the correct surface
- summarize recent work state

### Studio Pulse should not try to do in v3

- fake deep multi-agent orchestration
- present broad autonomous intelligence unsupported by system memory
- act like a generic social chatbot

### Execution actions

- `redesign` Studio Pulse around:
  - direction
  - review pressure
  - continuity warnings
  - next action
  - workload overview
- tie Studio Pulse responses to real counts, review events, asset coverage, and planning state
- remove any behavior that makes the surface feel performative rather than useful

### Exit criteria

- Studio Pulse gives operators materially useful operational guidance
- Studio Pulse feels like the place to decide what to do next
- Studio Pulse is clearly more than a stylized answer box

## Workstream 5: Character, identity, and continuity hardening

### Objective

Turn the strongest product idea into a real system advantage.

### Execution actions

- `keep` identity lock as a first-class concept
- `formalize` reference slots, never-change rules, and approved continuity anchors
- `formalize` Home System as environment and object continuity, not lifestyle clutter
- `redesign` character roles so each one has real decision territory
- `formalize` mode effects so they change system behavior in traceable ways

### Minimum v3 capability

Each character should have:

- a stable profile
- active identity lock rules
- reference assets
- explicit continuity anchors
- defined specialist role
- real mode consequences in prompts, review, or guidance

### Exit criteria

- Characters feel operational, not ornamental
- Identity lock meaningfully reduces drift
- Continuity data improves output quality instead of bloating prompts

## Workstream 6: UI hierarchy and runtime discipline

### Objective

Make the live product feel calm, coherent, and authoritative.

### Execution actions

- reduce flat navigation pressure
- standardize component hierarchy across core pages
- demote admin/system surfaces from primary operator prominence
- remove decorative UI density that hides core work
- reduce runtime ownership overlap on important surfaces
- enforce one owner per major page or subsystem where practical

### Exit criteria

- The core workflow feels visually and behaviorally unified
- Core pages look like parts of one product, not one shell plus many add-ons
- Support and system surfaces stop diluting the main experience

## Delivery sequence

## Phase 0: Stabilization contract

- Lock the v3 scope
- Define core versus support versus system surfaces
- Freeze speculative new first-class modules

## Phase 1: Product hierarchy reset

- Rework the navigation and surface classification
- Merge or demote weakly differentiated surfaces
- Clarify product language

## Phase 2: Artifact and state contracts

- Define ownership for prompts, outputs, assets, continuity, review, planner, and campaign artifacts
- Establish v3 source-of-truth rules

## Phase 3: Workflow closure

- Add lineage between prompt, output, review, and planning
- Introduce real review-event objects

## Phase 4: Studio Pulse upgrade

- Rebuild the surface logic around direction, review, and next action
- Tie guidance to real system state

## Phase 5: Character and continuity hardening

- Tighten character roles
- Formalize identity and continuity systems
- Ensure mode behavior has real effect

## Phase 6: UI and runtime cleanup

- Reduce visible fragmentation
- Stabilize core surface ownership
- Remove low-value noise

## Success metrics

### Product clarity

- Operators can explain the product’s job quickly and consistently
- Operators can identify the primary workflow without help

### Workflow usefulness

- A prompt can be traced to outputs, review, and planning implications
- Operators reuse prior learning instead of regenerating blindly

### Trust

- Operators know which surfaces are authoritative
- Fewer modules feel speculative or misleading

### Continuity quality

- Identity drift decreases
- Continuity references are applied selectively and effectively

### Runtime confidence

- Core surfaces have clear ownership
- Change risk on the live shell is reduced

## Non-goals

- Rewriting the whole frontend
- Building a full cloud-native backend
- Launching multi-agent orchestration
- Solving all analytics ambitions
- Turning every shell module into a real product immediately

## Final standard

v3 succeeds if it becomes a disciplined internal studio operating environment that people can trust daily. It does not need to feel futuristic. It needs to feel sharp, stable, and real.
